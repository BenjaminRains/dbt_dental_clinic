# Open Dental Airbyte Connector Proposal

**Status:** Draft  
**Last updated:** 2026-07-08  
**Supersedes:** [Open Dental Airbyte Connector Proposal.pdf](../../Open%20Dental%20Airbyte%20Connector%20Proposal.pdf)

---

## Goal

Build an open-source Airbyte source connector for Open Dental by extracting a reusable **Open Dental SDK** from the production MDC ETL pipeline.

The project should avoid becoming a massive rewrite. The first milestone is not “support everything.” The first milestone is:

> Prove that we can discover Open Dental tables, classify sync behavior, read selected tables incrementally with correct watermark semantics, and emit Airbyte-compatible records using a clean SDK layer.

---

## Context: Current MDC ETL Architecture

The MDC pipeline already solves Open Dental ingestion in production. The connector proposal should **extract proven read-side logic**, not re-invent it.

### Data flow today

```
OpenDental MySQL (source, readonly, VPN)
    ↓  EXTRACT — SimpleMySQLReplicator
Local MySQL (replication staging)
    ↓  LOAD — PostgresLoader
PostgreSQL analytics (schema: raw)
    ↓  TRANSFORM — dbt
staging → intermediate → marts
    ↓  PUBLISH (optional) — mdc publish analytics
Clinic RDS
```

### Sync model today

The pipeline uses **watermark-based incremental sync with upserts**, not MySQL binlog CDC.

| Concept | Implementation |
| --- | --- |
| Config source of truth | `tables.yml` generated by `analyze_opendental_schema.py` (analyzer v4.1) |
| Extraction strategies | `full_table`, `incremental`, `incremental_chunked` |
| Sync profiles | `append_only` vs `in_place_updates` |
| Watermark tracking | `etl_copy_status` (replication MySQL), `raw.etl_load_status` (PostgreSQL) |
| Writes | MySQL `INSERT ... ON DUPLICATE KEY UPDATE`; PostgreSQL `ON CONFLICT DO UPDATE` |
| Mutation tables | Timestamp watermarks (`DateTStamp`, `SecDateTEdit`) + optional 30-day lookback resync |
| Deletes | **Not propagated** (documented limitation) |

### What the SDK should carry forward

The SDK should encode the **read and classify** semantics that already work:

- Schema discovery and table metadata generation
- Incremental column detection and sync profile classification
- Query building with keyset pagination and lookback predicates
- Row cleaning and JSON-safe serialization
- Cursor / state management for watermark resume

### What stays in MDC ETL (not in the SDK)

The SDK is a **source reader**, not a full replication engine:

- Replication MySQL hop and `etl_copy_status` writes
- PostgreSQL loading (`PostgresLoader`, streaming upsert, COPY strategies)
- Airflow orchestration and nightly category processing
- dbt transforms and clinic RDS publish
- Layer 0 aggregate drift checks

An Airbyte deployment would replace the MySQL replication hop + custom loader with Airbyte destination sync. The SDK read logic should remain identical to what `SimpleMySQLReplicator` uses today.

---

## Design Principle

Do not make Airbyte depend directly on the current MDC ETL codebase.

Instead:

```
Current MDC ETL (production)
        ↓
Extract reusable read-side logic
        ↓
Open Dental SDK (standalone package)
        ↓
Airbyte source connector
```

The SDK should accept plain configuration dictionaries, not MDC-specific `Settings` or environment files.

## Relationship to Existing MDC Files

### Reuse from `etl_pipeline/etl_pipeline/core/connections.py`

**Extract:**

- `ConnectionFactory` — MySQL source connection lifecycle
- `ConnectionManager` — retry logic, rate limiting, connection validation

**Modify:**

- Accept Airbyte-style config dict (`host`, `port`, `database`, `username`, `password`)
- Remove `Settings` / `DatabaseType` dependency
- Do not configure replication or analytics target databases

### Reuse from `etl_pipeline/etl_pipeline/core/simple_mysql_replicator.py`

**Extract:**

- Batch sizing by performance category (tiny / small / medium / large)
- Full vs incremental strategy selection
- Keyset pagination
- Incremental cursor handling and watermark resolution
- Row cleaning and MySQL datetime normalization
- Lookback predicate wrapping for mutation tables

**Remove:**

- `target_engine` and replication MySQL writes
- `INSERT` / `UPSERT` / `DROP` / `CREATE` target table logic
- `etl_copy_status` dependency
- Postgres loader assumptions

### Reuse from `etl_pipeline/scripts/analyze_opendental_schema.py`

**Extract:**

- Table discovery (`discover_all_tables`) with exclusion patterns (`tmp_`, `temp_`, `backup_`, `#`, `test_`)
- Column discovery and primary key detection
- Incremental column detection (`find_incremental_columns`)
- Sync profile classification (`determine_sync_profile`)
- Watermark column selection (`select_primary_incremental_column`)
- Lookback resync config (`build_lookback_resync_config`)
- Extraction strategy determination (`determine_extraction_strategy`)
- Performance category and batch size assignment
- Schema drift detection (`compare_with_previous_schema`, schema hash)

**Modify:**

- Return Python objects / Airbyte catalog metadata instead of writing `tables.yml` to a project directory
- Accept plain config dict instead of `Settings` injection
- Expose `TableConfig` objects that mirror the `tables.yml` contract (see below)

### Reuse from `etl_pipeline/etl_pipeline/monitoring/replica_sync_config.py`

**Extract:**

- `get_replicator_watermark_column()`
- `get_loader_incremental_columns()` — rename for SDK context as incremental column resolution
- `build_mysql_lookback_predicate()` and `wrap_mysql_incremental_with_lookback_config()`
- `resolve_watermark_cursor()` — stale cursor invalidation when watermark column changes
- `build_mysql_loader_where_clause()` — shared WHERE clause builder

This module is the runtime contract between schema analysis and extraction. The SDK should own equivalent logic so Airbyte incremental sync matches MDC behavior.

## `tables.yml` Contract (SDK `TableConfig` Model)

The SDK `TableConfig` should mirror the fields the production pipeline already consumes. This keeps Airbyte streams aligned with MDC semantics.

```yaml
# Per-table config (representative)
patient:
  extraction_strategy: incremental          # full_table | incremental | incremental_chunked
  batch_size: 25000
  performance_category: small               # tiny | small | medium | large
  primary_key: [PatNum]
  incremental_columns: [PatNum, DateTStamp, SecDateTEdit]
  primary_incremental_column: DateTStamp
  incremental_strategy: or_logic            # or_logic | and_logic | single_column | none
  sync_profile: in_place_updates            # append_only | in_place_updates
  replicator_watermark_column: DateTStamp
  lookback_resync:                          # optional; mutation tables
    enabled: false
  estimated_rows: 12500
  estimated_size_mb: 4.2
  is_modeled: true
  schema_hash: abc123...
```

### Sync profile rules (from analyzer v4.1)

| Profile | When | Watermark column |
| --- | --- | --- |
| `append_only` | Log-style tables; modeled tables without a valid non-PK timestamp | Auto-increment PK acceptable for new-row capture |
| `in_place_updates` | Mutation seed table **and** valid timestamp in `incremental_columns` | `DateTStamp` or `SecDateTEdit` (first passing quality check) |

**Mutation seed tables** (conservative list, extend over time):

`procedurelog`, `claimproc`, `payment`, `adjustment`, `claim`, `patplan`, `paysplit`, `commlog`, `patient`

**Lookback resync** (30-day window on business date columns) applies to mutation tables where timestamp advancement is unreliable:

| Table | Predicate columns |
| --- | --- |
| `procedurelog` | `DateComplete`, `ProcDate` |
| `payment` | `PayDate` |
| `claimproc` | `DateCP`, `ProcDate` |
| `adjustment` | `AdjDate`, `ProcDate` |
| `claim` | `DateService`, `DateSent`, `DateReceived` |
| `paysplit` | `DatePay`, `ProcDate` |

### Known limitations to document in the connector

- Deletes are not captured (same as MDC ETL)
- `DateTStamp` advancement per table is not guaranteed by Open Dental schema
- PK-only tables classified as `append_only` may drift on in-place edits until full refresh
- See [ETL-FND-001](../findings/ETL-FND-001-replica-row-drift-procedurelog.md) and [ETL-FND-002](../findings/ETL-FND-002-sync-profile-pk-only-misclassification.md)

## Phased Delivery Plan

### Phase 0 — Project Setup

**Objective:** Create the repo/package skeleton without implementing the full connector.

**Deliverables:**

- `source-opendental` repo structure
- `pyproject.toml`, Dockerfile placeholder, basic README
- Initial SDK and Airbyte package stubs
- `pytest` structure

**Success criteria:**

- `pytest` runs
- Package imports cleanly
- Docker build starts
- README explains project purpose and relationship to MDC ETL

---

### Phase 1 — Open Dental SDK Core

**Objective:** Build a standalone SDK that connects to Open Dental MySQL and inspects the database.

**Deliverables:**

- `opendental_sdk/config.py`
- `opendental_sdk/connection.py`
- `opendental_sdk/schema_discovery.py`
- `opendental_sdk/table_config.py`

**Features:**

- Validate config
- Connect to MySQL and check connection
- List tables (with exclusion patterns)
- List columns, detect primary keys
- Detect row counts and table sizes
- Classify `sync_profile` and extraction strategy
- Emit `TableConfig` objects matching `tables.yml` contract

**Success criteria:**

- SDK connects to Open Dental without MDC `Settings`
- SDK discovers ~446 tables (current production count)
- SDK returns table metadata as Python objects
- `procedurelog` classified as `in_place_updates` with `DateTStamp` watermark

---

### Phase 2 — Query Builder and Row Reader

**Objective:** Read rows from selected Open Dental tables without writing to a destination.

**Deliverables:**

- `opendental_sdk/query_builder.py`
- `opendental_sdk/cursor.py`
- `opendental_sdk/row_cleaner.py`

**Features:**

- Full table reads
- Batch reads with performance-category batch sizes
- Keyset pagination
- Incremental reads with watermark cursors
- Lookback predicate wrapping for mutation tables
- Stale cursor invalidation when watermark column changes
- Row serialization to JSON-compatible dicts
- MySQL datetime/time cleanup
- Safe identifier quoting

**Initial tables** (prioritized by production usage and mutation risk):

| Table | Sync profile | Why first |
| --- | --- | --- |
| `patient` | `in_place_updates` | Core dimension; demographic edits |
| `appointment` | `append_only` | High-volume scheduling |
| `procedurelog` | `in_place_updates` + lookback | Proven drift case (ETL-FND-001) |
| `provider` | `append_only` | Small reference table |
| `payment` | `in_place_updates` + lookback | AR / revenue |
| `paysplit` | `in_place_updates` + lookback | Payment allocation |
| `claim` | `in_place_updates` + lookback | Insurance |
| `claimproc` | `in_place_updates` + lookback | Insurance procedures |

**Success criteria:**

- Fetch `patient` rows in batches
- Fetch `procedurelog` with keyset pagination and lookback OR predicate
- Resume incremental read from a cursor value
- Invalidate cursor when watermark column changes (e.g. `ProcNum` → `DateTStamp`)
- Rows serialize cleanly to JSON-compatible dicts

---

### Phase 3 — Airbyte Minimal Connector

**Objective:** Wrap the SDK in an Airbyte source connector.

**Deliverables:**

- `source_opendental/source.py`
- `source_opendental/streams.py`
- `source_opendental/spec.json`

**Airbyte methods:**

- `check_connection()`
- `discover()`
- `read()`

**Initial behavior:**

- Discover selected tables with Airbyte catalog metadata
- Emit Airbyte `RECORD` messages
- Emit Airbyte `STATE` messages (per-stream watermark state)
- Support full refresh
- Support incremental sync with sync-profile-aware watermarks

**Success criteria:**

- Airbyte can run `check`
- Airbyte can discover tables
- Airbyte can sync `patient` (incremental, timestamp watermark)
- Airbyte can sync `appointment` (incremental)
- Airbyte can sync `procedurelog` incrementally with lookback predicate

### Phase 4 — Schema Drift Handling

**Objective:** Add Open Dental version/schema-change resilience.

**Deliverables:**

**Features:**

- Detect added columns

**Success criteria:**

- Connector survives non-breaking column additions

- Connector documents required fields per stream
- Behavior matches MDC schema analysis changelog output

### Phase 7 — Optional Product Layer

**Objective:** Build on top of ingestion.

The MDC repo already contains a dbt package (`dbt_dental_models/`). The connector + existing dbt models is the real differentiator.

**Possible extensions:**

- Publish Open Dental dbt package separately (staging models already exist: `stg_opendental__*`)
- Patient journey, treatment acceptance, insurance claim marts (partially built)
- Power BI reference dashboard
- Snowflake / Databricks deployment guides
- Migration guide: MDC ETL → Airbyte + existing dbt models

**Long-term positioning:**

- The **connector** gets data out of Open Dental with correct sync semantics.
- The **dbt package** and analytics models make the data useful.
- That combination is the real differentiator.

## Practical First Sprint

### Sprint 1 scope

Do not build the Airbyte connector yet. Build only:

- `opendental_sdk/config.py`
- `opendental_sdk/connection.py`
- `opendental_sdk/schema_discovery.py`
- `opendental_sdk/table_config.py`
- `opendental_sdk/query_builder.py`
- `opendental_sdk/row_cleaner.py`

### Sprint 1 goal

Run this locally:

```python
from opendental_sdk import OpenDentalClient

client = OpenDentalClient.from_config({
    "host": "10.x.x.x",
    "port": 3306,
    "database": "opendental",
    "username": "readonly_user",
    "password": "...",
})

client.check_connection()

tables = client.discover_tables()
procedurelog = tables["procedurelog"]
assert procedurelog.sync_profile == "in_place_updates"
assert procedurelog.replicator_watermark_column == "DateTStamp"

for row in client.read_table("procedurelog", batch_size=1000, mode="incremental"):
    print(row)
```

### Sprint 1 validation against MDC ETL

Compare SDK output to production pipeline for the initial eight tables:

| Check | MDC reference |
| --- | --- |
| Table metadata | `etl_pipeline/etl_pipeline/config/tables.yml` |
| Incremental WHERE clause | `replica_sync_config.build_mysql_loader_where_clause()` |
| Row shape after cleaning | `SimpleMySQLReplicator` row output |
| Sync profile classification | `analyze_opendental_schema.determine_sync_profile()` |

Once the SDK reads match MDC extraction semantics, Airbyte becomes a wrapper problem instead of a giant unknown.

## Open Questions

1. **Monorepo vs separate repo:** Extract SDK into `source-opendental/` within this repo first, or publish as a standalone GitHub repo from day one?
2. **State storage:** Airbyte per-stream state vs SDK state format — should SDK state serialize the same fields as `etl_copy_status` for migration compatibility?
3. **Full table scope:** Ship with all ~446 tables discoverable but only ~20 streams enabled by default?
4. **Lookback in Airbyte:** Encode lookback as part of incremental WHERE in the stream, or as a separate "resync" mode?
